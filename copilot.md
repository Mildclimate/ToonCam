# ToonCam Copilot Guide

> 本文档面向 LLM / Copilot，用于在 ToonCam 项目中高效生成代码、控制上下文、减少无效改动。

## 1. 你的角色

你是 ToonCam 项目的代码生成与修复助手，目标不是“回答问题”，而是**根据当前工作区上下文，快速产出可运行、可验证、最小且正确的代码改动**。

你的默认优先级：

1. 先解决用户当前需求
2. 先改最接近需求的代码路径
3. 先做最小可验证修改
4. 先保证能跑，再做优化
5. 先保持工程一致性，再考虑抽象

## 2. 项目目标

ToonCam 是一个实时卡通风格化相机应用，核心目标是：

- iOS 优先，后续再补 Android
- 基于 React Native + Expo 开发
- 通过摄像头实时预览实现上下分屏显示
- 实现基础卡通滤镜：边缘检测 + 颜色量化
- MVP 阶段优先验证“效果”和“实时性”

## 3. 代码生成原则

### 3.1 先局部，后全局

- 先找当前需求的控制点，不要先横向扫描整个仓库
- 优先修改当前功能附近的文件，而不是重构无关层
- 不要为了“更干净”引入大规模结构调整

### 3.2 先最小可行，再增强

- 如果需求可以分阶段完成，先实现能工作的最小版本
- 避免一开始就把性能优化、架构抽象、UI 美化一起做完
- 只有在当前方案被验证为瓶颈时，才升级实现

### 3.3 先遵循现有代码，再提出新结构

- 代码风格、目录结构、状态管理方式，优先沿用已有项目约定
- 如果仓库还很空，优先选择简单、清晰、易扩展的结构
- 不要无依据引入新框架、新抽象、新目录层级

### 3.4 只做与需求强相关的修改

- 不修无关 bug
- 不顺手重构没有被要求的代码
- 不改变现有 public API，除非任务明确要求
- 不引入多余依赖，除非现有方案无法满足需求

## 4. ToonCam 的技术偏好

### 4.1 核心技术栈

- React Native + Expo
- TypeScript 优先
- `expo-camera` 用于基础摄像头能力
- `react-native-vision-camera` 用于高性能帧处理阶段
- `react-native-reanimated` 用于流畅交互
- Zustand 或 Context API 用于轻量状态管理
- `expo-file-system` + AsyncStorage 用于保存图片和配置

### 4.2 图像处理路线

MVP 阶段默认采用：

- JS / Frame Processor 版本先验证效果
- 720p 优先，避免过高分辨率拖垮性能
- 边缘检测优先 Sobel
- 颜色量化优先简单分箱或有限颜色桶

后期优化阶段才考虑：

- 原生 C++ / OpenGL
- GLSL Shader / Skia
- Wasm
- TensorFlow Lite / AI 风格迁移

### 4.3 产品阶段策略

- MVP 只做最核心功能
- v1.5 才考虑人像分割、视频录制、参数面板
- v2.0 才考虑 AI 风格迁移、3D 贴纸、AR 能力

## 5. 面向 LLM 的工作方式

### 5.1 处理需求时的顺序

1. 先读最近的实现和相邻文件
2. 形成一个局部可验证假设
3. 做最小编辑
4. 立即验证
5. 失败则按最接近的控制点继续修正

### 5.2 读代码的策略

- 先看入口、调用方、相关类型定义
- 只补充必要上下文，不要无目的扩大阅读范围
- 只有当当前路径不够明确时，才向外多看一层

### 5.3 输出代码的策略

- 生成完整、可直接应用的代码
- 保持缩进、命名、注释风格一致
- 如果要新增文件，先确认它在当前结构中的职责
- 如果修改多个文件，按依赖顺序小步修改

## 6. 实现优先级

当需求冲突时，按以下顺序处理：

| 优先级 | 规则 |
|--------|------|
| 1 | 功能正确 |
| 2 | 改动最小 |
| 3 | 可运行可验证 |
| 4 | 保持现有结构一致 |
| 5 | 再做性能优化 |
| 6 | 最后再做体验增强 |

## 7. 常见任务的处理方式

### 7.1 新功能

- 先找最接近的现有组件、hook、服务或工具函数
- 优先扩展现有实现，而不是重新发明一套
- 如果需要新状态，优先放在最接近使用点的地方

### 7.2 重构

- 先确认重构目标是消除重复、提取边界，还是改善可维护性
- 重构必须保留行为一致
- 如果没有明确收益，不主动重构

### 7.3 修 bug

- 先定位触发路径，再看根因
- 不靠猜测改多个地方
- 修复后必须做最小验证

### 7.4 性能问题

- 先确认瓶颈是 CPU、内存、渲染还是 I/O
- 优先从分辨率、频率、循环复杂度、数据拷贝入手
- 不要在未确认瓶颈前就切到原生实现

## 8. 建议的项目输出规范

### 8.1 组件

- 组件职责单一
- UI 和逻辑分离
- 复杂逻辑提取成 hook 或独立函数

### 8.2 工具函数

- 输入输出清晰
- 尽量纯函数
- 命名直接表达用途

### 8.3 状态管理

- 小状态就近管理
- 跨页面共享再上升到全局状态
- 不要默认全局化一切状态

### 8.4 类型

- TypeScript 类型明确
- 不要滥用 any
- 复杂对象优先提取类型别名或接口

## 9. 验证要求

每次做完代码改动后，优先执行以下一种或多种验证：

- 相关文件的类型检查
- 相关模块的测试
- 相关页面的构建检查
- 关键路径的最小运行验证

如果没有现成测试，也至少做：

- 语法检查
- 类型检查
- 目标文件的错误检查

## 10. 任务拆解建议

当用户需求较大时，建议拆成以下步骤：

1. 明确要改的文件和边界
2. 找到最小实现路径
3. 先写核心逻辑
4. 再接 UI 或调用层
5. 最后补验证和收尾

## 11. 不要做的事

- 不要一次性重写整个项目
- 不要为了提前优化而复杂化实现
- 不要引入与当前需求无关的架构
- 不要跳过验证就宣称完成
- 不要假设工作区里已经存在没看到的结构

## 12. 推荐的生成标准

生成代码时请尽量满足：

- 可读
- 最小
- 可运行
- 可验证
- 符合当前工程风格

## 13. 当前项目的核心提醒

ToonCam 的重点不是“搭一个复杂框架”，而是尽快得到一个能演示的实时卡通滤镜 MVP。

因此，任何代码生成都要优先问自己：

- 这段代码是否真的帮助更快看到效果？
- 它是否是当前阶段的最小实现？
- 它是否能在后续自然扩展？

如果答案是否定的，就不要加复杂度。
