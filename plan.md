## 一、项目背景与定位

### 1.1 一句话定位

一款实时将摄像头画面转为卡通风格的相机应用，上下分屏显示真实与风格化画面。

### 1.2 核心价值

| 维度 | 说明 |
|------|------|
| **作品集价值** | 融合动画审美、前端工程、图形渲染、AI 能力于一个可展示的应用 |
| **技术证明力** | 面试时直接打开 App 演示，比口头描述有说服力 10 倍 |
| **迭代空间** | 从"卡通滤镜"可逐步扩展到贴纸、3D 角色驱动、AI 风格迁移 |

### 1.3 竞品参考

| 竞品 | 特点 | 差距 / 启发 |
|------|------|-------------|
| **剪映 / CapCut** | 成熟但闭源，特效丰富 | 功能全但不可定制 |
| **Prisma** | 静态风格迁移，非实时 | 无实时能力 |
| **ToonMe** | 静态卡通化，非实时 | 无实时能力 |
| **Snapchat** | 实时但滤镜质量参差不齐 | 可参考其交互模式 |

> **差异化定位：** 开源、实时、上下屏对比、聚焦卡通风格质量。

## 二、技术选型

### 2.1 整体技术栈

| 维度 | 选择 | 理由 |
|------|------|------|
| **跨平台框架** | React Native + Expo | 熟悉 JS/TS 生态，一套代码跑 iOS/Android，Expo 简化原生配置 |
| **摄像头采集** | `expo-camera` | Expo 官方库，稳定可靠 |
| **图像处理** | `react-native-vision-camera` + Frame Processors | 高性能帧处理，支持在 JS/原生层实时操作像素 |
| **图像处理核心** | 自研 JS 图像处理管道（Canvas 风格像素操作）→ 后期迁移到原生（C++/Shader） | 先快速验证效果，再优化性能 |
| **UI 框架** | React Native 原生组件 + `react-native-reanimated` | 流畅的 UI 交互 |
| **状态管理** | Zustand 或 Context API | 轻量，避免过度工程化 |
| **存储** | `expo-file-system` + AsyncStorage | 保存图片 / 参数配置 |

### 2.2 图像处理方案选型（关键决策）

| 方案 | 优点 | 缺点 | 适用阶段 |
|------|------|------|----------|
| **纯 JS 处理（Frame Processors）** | 快速实现、易于调试、无需原生编译 | 性能有限，高分辨率可能卡顿 | **MVP 阶段** |
| **原生 C++ / OpenGL 处理** | 性能最优，可复用图形学知识 | 编译周期长、调试难度大 | 后期优化阶段 |
| **GLSL Shader（VisionCamera 的 Skia 集成）** | GPU 加速，性能好 | 学习曲线陡峭 | 进阶版本 |

> **决策：** 先走方案 1，验证效果和用户反馈。一旦发现性能瓶颈，再逐步迁移到方案 2/3。

### 2.3 为什么用 Vision Camera 而非 `expo-camera` 直接处理

`react-native-vision-camera` 提供了 **Frame Processors** 能力，允许你在每一帧上运行 JS 函数来修改像素数据，且性能优化优于纯 JS 方案。

- ✅ 支持直接在帧上操作 `Uint8Array`，是实现像素级滤镜的基础
- ✅ 生态成熟，社区有大量滤镜示例可供参考

## 四、技术挑战与对策

| 挑战 | 描述 | 对策 |
|------|------|------|
| **性能瓶颈** | 在 Frame Processor 中逐帧处理像素，JS 处理能力有限，高分辨率下容易掉帧 | 优先降至 720p 处理，后期使用 `react-native-reanimated` 或原生 C++/OpenGL Shader |
| **滤镜效果调试** | 在手机上调试图像处理算法，无法像 Web 端用 Chrome DevTools 那样快速迭代 | 先用模拟器开发，用 `console.log` 打印像素数据；后期用静态图片调试，验证算法后再接摄像头 |
| **原生依赖兼容性** | Vision Camera 等库可能因 iOS 版本或 Expo 版本兼容问题无法编译 | 用最新的 Expo SDK（如 50+），严格按官方文档配置，遇到问题先查 GitHub Issue |
| **内存占用** | 连续处理高清帧可能导致 App 内存暴涨 | 使用 `PixelFormat.yuv` 格式减少内存占用；及时释放帧数据；不上高分辨率功能（如 4K 录制） |

## 七、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|:----:|:----:|------|
| Vision Camera 在 iOS 上编译失败 | 🟡 中 | 🔴 高 | 备选方案：使用 `expo-camera` + 后台处理（性能更差但可用） |
| 图像处理 JS 性能远低于预期 | 🔴 高 | 🔴 高 | 降低处理分辨率至 480p，先保证功能可用；同时研究 `react-native-skia` 方案 |
| 滤镜效果不好看 | 🟡 中 | 🟡 中 | 多参考经典卡通风格参数；在 GitHub 找开源实现参考；可先实现 3 种以上风格供对比 |
| 项目过大导致拖延，最后不了了之 | 🟡 中 | 🔴 高 | **严格控制 MVP 范围，不加功能**；每周做一次"可演示"增量 |
